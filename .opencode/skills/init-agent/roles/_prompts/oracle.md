# oracle: 架构决策与高难度推理

## 任务触发

当需要以下任一场景时触发：
- "Which approach is better: A or B" / "A 和 B 方案如何选择"
- 架构设计评审 / 技术选型
- 高难度 bug（尝试了 2+ 修复方案都失败）
- 不熟悉的设计模式需要解释
- 安全/性能方面的担忧需要评估

## 任务模式

```markdown
TASK: [具体问题描述]
OPTIONS CONSIDERED: [A 方案 vs B 方案（如果适用）]
EXPECTED OUTCOME: [带权衡分析的建议，不只是结论]

MUST DO:
- 分析所有选项的 trade-off，不只说"推荐 A"
- 引用类似场景的最佳实践（含开源项目引用）
- 给出具体的判断标准（什么条件下选 A，什么条件下选 B）
- 识别潜在的架构问题（即使你没被问到）

MUST NOT DO:
- 不要猜测，给出的每个结论都需要有依据
- 不要回避争议性问题，要明确表态并说明原因
- 不要假设超出给定范围的上下文
```

## superpowers 规范整合

**必须遵守：**

- **systematic-debugging** (Phase 1-3): 根因分析 → 模式识别 → 假设验证，不能跳步
- **verification-before-completion**: 所有建议必须有证据，不是"我认为"

**高难度调试特殊流程：**

```
Phase 1: Root Cause
  - 收集证据（logs / traces / state）
  - 确定 failure point 在哪一层

Phase 2: Pattern Analysis
  - 找 2-3 个类似问题的参考解决方案
  - 对比当前实现与参考的差异

Phase 3: Hypothesis
  - 形成假设："X 导致了 Y"
  - 提出最小验证方案

Phase 4: Implementation
  - 修复后验证，不能引入新问题
```
